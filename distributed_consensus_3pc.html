Three-Phase Commit (3PC)

Non-Blocking Consensus Algorithm
Message Complexity: O(3n)

Algorithm Overview

The Three-Phase Commit (3PC) protocol is an enhanced distributed consensus algorithm that solves the blocking problem of 2PC by adding a "pre-commit" phase. This protocol ensures that all nodes in a distributed system either commit or abort a transaction consistently, even in the presence of coordinator failures.

3PC operates in three phases: CanCommit where participants express readiness, PreCommit where the coordinator confirms all are ready, and DoCommit where the actual commit occurs. The additional phase allows participants to make progress independently if the coordinator fails after the pre-commit phase.

Key Feature

Non-blocking protocol that handles coordinator failures

Performance

Requires 3 round trips but provides better fault tolerance

Fault Tolerance

Handles coordinator failures without indefinite blocking

Approach

Coordinator-driven with intermediate pre-commit state

Algorithm Steps

The Three-Phase Commit protocol operates in three distinct phases:

Phase 1: CanCommit

The coordinator sends a CanCommit? message to all participants. Each participant votes YES if it can commit, or NO if it cannot. Participants lock resources during this phase.

Phase 2: PreCommit

If all participants vote YES, the coordinator sends PreCommit messages. Participants acknowledge and transition to the pre-commit state. This state indicates they are ready to commit but haven't committed yet.

Phase 3: DoCommit

The coordinator sends DoCommit messages to all participants. Participants commit the transaction and send acknowledgments. If any participant voted NO in Phase 1, the coordinator sends Abort messages.

Failure Handling

If the coordinator fails after PreCommit, participants can autonomously decide to commit after a timeout. This non-blocking behavior is the key advantage over 2PC.

Pseudocode

// Coordinator Logic
function coordinator():
  // Phase 1: CanCommit
  send CanCommit? to all participants
  wait for responses
  if all votes == YES:
      // Phase 2: PreCommit
      send PreCommit to all
      wait for ACKs
      // Phase 3: DoCommit
      send DoCommit to all
  else:
      send Abort to all
  wait for ACKs
  complete transaction

// Participant Logic
function participant():
  on CanCommit? received:
      if canCommit():
          lockResources()
          send YES to coordinator
      else:
          send NO to coordinator
          
  on PreCommit received:
      transition to PRECOMMIT state
      send ACK to coordinator
      
  on DoCommit received:
      commitTransaction()
      releaseLocks()
      send ACK to coordinator
      
  on Abort received:
      abortTransaction()
      releaseLocks()
      send ACK to coordinator
      
  // Timeout handling
  if timeout in PRECOMMIT state:
      if majority in PRECOMMIT:
          commitTransaction()
      else:
          abortTransaction()

Implementation

Below is a simplified Python implementation of the Three-Phase Commit protocol. The implementation includes coordinator and participant logic with timeout-based failure handling.

Python Implementation 
import random
import threading
import time

class Coordinator:
    def __init__(self, participants):
        self.participants = participants
        self.state = "INIT"
    
    def start_transaction(self):
        # Phase 1: CanCommit
        self.state = "CAN_COMMIT"
        votes = []
        for p in self.participants:
            vote = p.can_commit()
            votes.append(vote)
        
        if not all(vote == "YES" for vote in votes):
            self.state = "ABORT"
            for p in self.participants:
                p.abort()
            return "ABORTED"
        
        # Phase 2: PreCommit
        self.state = "PRE_COMMIT"
        acks = []
        for p in self.participants:
            ack = p.pre_commit()
            acks.append(ack)
        
        if not all(ack == "ACK" for ack in acks):
            self.state = "ABORT"
            for p in self.participants:
                p.abort()
            return "ABORTED"
        
        # Phase 3: DoCommit
        self.state = "DO_COMMIT"
        for p in self.participants:
            p.do_commit()
        
        # Wait for final ACKs
        final_acks = []
        for p in self.participants:
            final_acks.append(p.ack())
        
        if all(ack == "ACK" for ack in final_acks):
            self.state = "COMPLETE"
        return self.state

class Participant:
    def __init__(self, id):
        self.id = id
        self.state = "READY"
        self.vote = None
        self.timeout_thread = None
    
    def can_commit(self):
        # 90% success rate
        if random.random() > 0.1:
            self.state = "WAITING"
            self.vote = "YES"
        else:
            self.state = "ABORTED"
            self.vote = "NO"
        return self.vote
    
    def pre_commit(self):
        if self.state == "WAITING":
            self.state = "PRECOMMIT"
            # Start timeout for coordinator failure
            self._start_timeout()
        return "ACK"
    
    def do_commit(self):
        if self.state == "PRECOMMIT":
            self.state = "COMMITTED"
            self._stop_timeout()
        return "ACK"
    
    def abort(self):
        self.state = "ABORTED"
        self._stop_timeout()
        return "ACK"
    
    def ack(self):
        return "ACK"
    
    def _start_timeout(self):

        self.timeout_thread = threading.Thread(target=timeout_handler)
        self.timeout_thread.start()

Applications

The Three-Phase Commit protocol is used in systems requiring high availability and fault tolerance:

  • Distributed databases requiring high availability
  • Financial transaction systems
  • Cloud-based distributed systems
  • Systems with unreliable networks
  • Real-time collaborative applications

3PC vs 2PC

Key differences between Three-Phase and Two-Phase Commit:

  • Blocking: 3PC is non-blocking, 2PC blocks on coordinator failure
  • Phases: 3PC has an additional pre-commit phase
  • Fault Tolerance: 3PC handles coordinator failures better
  • Performance: 2PC has lower latency (2 round trips vs 3)
  • Complexity: 3PC is more complex to implement
